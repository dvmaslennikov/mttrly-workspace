## Analyst Notes & Refined Recommendations

### Key Findings

1. **Most queries work GREAT.** 18 out of 23 queries have 30%+ signal. The original concern about "too generic queries returning garbage" is largely solved by using exact-phrase matching (`"..."`) with domain-specific terms.

2. **The biggest junk source is `@bankrbot`** ‚Äî a crypto token deployment bot that uses "deploy failed" language. Add `-bankrbot -"deploy the token" -"on Base" -"fee beneficiary" -web3 -"on-chain"` to pain_points queries.

3. **Three queries should be REMOVED or reworked:**
   - `B1: "vibe coding" ship OR deploy OR launch OR built` ‚Äî 87% junk. "vibe coding" is now mainstream discussion, not a deploy-focused term. The unqualified OR terms match everything.
   - `B3: "just shipped" side project OR indie OR solo` ‚Äî 87% junk. "just shipped" matches product launches, app store submissions, etc., not deploy pain.
   - `D2: "just want to ship" -hiring -job` ‚Äî 83% junk. Too vague, captures general product aspirations.

4. **A7: "prod is down" / "production incident" / "deployment failed"** ‚Äî 33% signal, 20/30 junk. The problem is `"deployment failed"` matching crypto bots. Replace with `"deployment failed" -token -"on Base" -bankrbot -web3`.

5. **Freshness is the main bottleneck.** Many high-signal queries have few fresh tweets:
   - A4 (hours debugging): 0 tweets < 48h ‚Äî this is a long-tail query, run weekly not daily
   - A5 (hate devops): 1 tweet < 48h ‚Äî same, weekly cadence
   - A6 (deployment hell): 5 tweets < 48h ‚Äî can do every-other-day
   - C4 (leaving heroku): 0 tweets < 48h ‚Äî weekly

6. **Replies are GOLD:** 83% of replies (197/238) are relevant. This is because reply threads around deploy topics stay on-topic. **Always include replies.**

7. **Like threshold: likes >= 0 for daily scan, likes >= 3 for digest.** At likes >= 1, you keep 186 relevant tweets out of 246 total. At likes >= 3, you get 81 quality tweets. For a digest of 5-15 best tweets, likes >= 3 is optimal.
